# Atlas RAG 项目面试讲解稿

## 1. 30 秒项目介绍

Atlas 是我做的一个智能知识库与 RAG 问答系统，后端主要使用 Spring Boot、MyBatis-Plus、MySQL 和 JWT 鉴权实现。用户可以创建笔记本并保存 Markdown 笔记，系统会在保存后异步进行文本切片和 embedding 生成，然后支持关键词检索、语义检索、RRF 混合检索，以及基于召回片段的 RAG 问答。

这个项目的重点不是单纯调用大模型 API，而是把一个后端系统中常见的用户鉴权、数据建模、异步任务、接口抽象、检索排序和降级兜底串成一个完整闭环。

项目描述：

- 使用 Spring Boot、MyBatis-Plus、MySQL 实现用户认证、笔记本、笔记、标签和知识片段管理模块

## 3. 面试官可能问：你这个项目怎么跑起来？

回答：

项目分成后端和前端两部分。后端是 Spring Boot，默认端口 8080，数据库用 MySQL。前端是 Vue 3 和 Vite。

启动流程是：

1. 先创建 MySQL 数据库 atlas，并执行后端的 schema.sql 初始化表结构
2. 启动后端：进入 atlas-backend，运行 mvnw.cmd spring-boot:run
3. 启动前端：进入 atlas-frontend，运行 npm run dev
4. 在页面注册登录，创建笔记本，保存笔记
5. 保存笔记后后端异步生成 chunk 和 embedding
6. 然后可以执行语义检索、混合检索和 RAG 问答

为了方便演示，我还准备了 docs/demo.http，可以直接通过 HTTP Client 插件调用注册、登录、创建笔记、搜索和 RAG 问答接口。

## 4. 面试官可能问：RAG 流程是什么？

回答：

我的流程分为写入和查询两部分。

写入阶段：

1. 用户保存 Markdown 笔记
2. NoteService 保存笔记主体数据
3. 保存成功后触发 EmbeddingPipeline
4. Pipeline 先删除这篇笔记旧的 chunk
5. TextChunker 按段落和句子切分文本，并保留 overlap
6. 每个 chunk 写入 note_chunk 表
7. 异步线程池调用 EmbeddingClient 生成向量
8. 向量保存到 MySQL，同时如果启用了 Qdrant，也会 upsert 到 Qdrant

查询阶段：

1. 用户输入问题
2. 系统对 query 生成 embedding
3. 语义检索召回相似 chunk
4. 混合检索时再融合关键词检索结果
5. ChatService 把 topK 片段和问题组装成 prompt
6. ChatClient 调用真实模型或本地兜底回答器生成答案
7. 返回答案和引用片段

## 5. 面试官可能问：为什么要异步生成 embedding？

回答：

因为 embedding 生成可能比较慢，尤其是接外部 API 或者一次保存长文档时，直接同步生成会导致保存接口响应很慢，用户体验不好。

所以我的设计是：保存笔记接口只负责把核心数据落库，成功后把切片和向量生成交给异步线程池。这样保存接口可以快速返回，后端慢慢补齐索引。

同时我给异步任务做了重试机制，如果单个 chunk embedding 失败，会最多重试 3 次，降低临时网络波动或模型服务异常带来的影响。

## 6. 面试官可能问：TextChunker 怎么切？

回答：

我没有简单按固定长度硬切，而是先按空行分段，再按中文和英文标点做句子级切分。目标 chunk 长度大约是 500 字符，并保留 50 字符 overlap。

这样做的原因是：

1. 尽量保持语义完整，避免一句话被切断
2. overlap 可以减少上下文断裂，比如答案相关信息刚好跨 chunk 边界时仍有机会被召回
3. chunk 不宜太大，否则检索粒度粗；也不宜太小，否则上下文不足、召回片段太碎

## 7. 面试官可能问：语义检索和关键词检索有什么区别？

回答：

关键词检索更适合精确词匹配，比如查“JWT”“线程池”“Qdrant”这类明确关键词。语义检索更适合表达不完全一致的查询，比如用户问“系统怎么做离线兜底”，笔记里可能写的是“未配置外部模型时使用本地 HashEmbeddingClient”，这种就需要语义相似度。

所以我做了 hybrid search，把关键词和语义两路结果融合起来。这样既能保留关键词检索的精确性，也能利用语义检索的泛化能力。

## 8. 面试官可能问：RRF 是什么？

回答：

RRF 是 Reciprocal Rank Fusion，用于融合多个排序列表。它不直接依赖原始分数，因为关键词检索分数和向量相似度分数不一定在同一个尺度上。

计算方式大致是：

score += 1 / (k + rank)

rank 越靠前贡献越大，k 通常取 60 左右，用来平滑不同排序结果的影响。

我在项目里用 RRF 融合关键词检索和语义检索结果，避免直接把两个不同分布的分数相加。

## 9. 面试官可能问：没有真实大模型 API，你怎么演示？

回答：

我做了接口抽象和本地兜底。

EmbeddingClient 有本地 HashEmbeddingClient，如果没有配置真实 embedding 模型，就用本地 hash 方式生成 256 维向量，保证语义检索流程能跑通。

ChatClient 也有 LocalRagChatClient，如果没有配置真实 Chat 模型，就根据召回片段生成一个简单中文答案，并返回引用来源。这样 MVP 演示不依赖外部 API Key，也不怕网络问题。

但我会明确说明：本地 hash embedding 和本地回答器是开发演示兜底，不是生产方案。生产环境会切换到真实 embedding 模型和大语言模型。

## 10. 面试官可能问：这个项目跟后端开发有什么关系？

回答：

虽然项目主题是 AI/RAG，但我主要展示的是后端工程能力：

1. 用户认证：Spring Security + JWT
2. 数据建模：用户、笔记本、笔记、标签、chunk、AI 模型配置
3. 接口设计：RESTful API 和统一响应结构
4. 异步任务：embedding 生成通过线程池异步执行
5. 接口抽象：EmbeddingClient、VectorStoreClient、ChatClient 便于替换实现
6. 容错降级：没有外部模型、没有 Qdrant 时仍能跑通 MVP
7. 检索排序：关键词、语义、RRF 混合检索
8. 前后端联调：Vue 页面可以完成完整演示闭环

所以它不是只会调 API，而是一个后端系统加 AI 能力的工程实践。

## 11. 面试官可能问：Qdrant 在项目里怎么用？

回答：

Qdrant 是可选向量数据库。启用时，chunk 的 embedding 会 upsert 到 Qdrant collection 中，语义检索优先走 Qdrant。

但为了降低本地演示依赖，我默认把 Qdrant 关闭。关闭时，向量仍然存在 MySQL 的 note_chunk 表中，查询时在 JVM 里计算 cosine similarity。

这样设计的好处是：

1. MVP 更容易跑起来
2. 面试演示不依赖 Docker 或外部服务
3. 后续生产化时可以打开 Qdrant，提高大规模数据下的检索性能

## 12. 面试官可能问：项目现在有什么不足？

回答：

目前它是 MVP，还有一些可以继续优化的地方：

1. 本地 hash embedding 只能用于演示，语义质量不如真实 embedding 模型
2. Qdrant 默认关闭，数据规模变大后 JVM 内计算相似度性能会下降
3. RAG 问答暂时没有做多轮对话记忆
4. chunk 删除和向量库删除还可以进一步统一事务或补偿机制
5. 可以增加召回评估，比如 Hit Rate、MRR，或者对比不同 chunk 参数的效果

我后续会优先优化真实 embedding 接入、Qdrant 异常兜底、多轮问答上下文和检索质量评估。
